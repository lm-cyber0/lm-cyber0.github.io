Zero Touch Connector Migration Automation

Migrated a large distributed fleet of multi-cloud AWS connectors from manually configured per-account scan schedules to Qualys's Zero Touch scanning feature via the Qualys API. The interesting engineering problem wasn't the migration itself; it was the retrofit. The fleet already had years of accumulated manual scan configurations that needed to be reconciled against Zero Touch enablement state before anything could be decommissioned. Built comparison automation that diffed the two states and programmatically resolved the delta at scale, retroactively converting all legacy manual scans without dropping coverage. After cutover, new accounts get coverage automatically when provisioned; decommissioned ones drop off without intervention. Connector provisioning time: days to seconds. Recurring manual scan operations: eliminated.

Related: built a separate audit reporting pipeline using the Qualys and Confluence APIs with pandas to aggregate scan data into structured, auditor-ready reports, scoped to avoid overreaching platform permissions and giving auditors direct access to the data they needed without pulling in the security team for every request.

WhisperLink AI

Self-hosted AI voice assistant designed and built end-to-end. The client is a Raspberry Pi 4 with a ReSpeaker 2-Mic HAT for audio capture and dual TFT displays for status and output. It connects over WebSocket to a GPU inference server running faster-whisper for speech-to-text, Ollama (llama3.1) for LLM inference, and Piper for text-to-speech, all running on local hardware. No data leaves the network at any point. The FastAPI backend is hardened with TLS, API key authentication, and rate limiting. The system is fully functional as a practical daily-use assistant and doubles as a hands-on exercise in designing a secure, self-hosted AI stack from the network layer up rather than stitching together cloud services.

FlipperTwo Maker Edition WIP

Dual-processor wireless security multi-tool built on a LilyGo T-Display-S3 (ESP32-S3) as the main processor with an ESP32-C6 co-processor handling wireless attacks over a UART command protocol. The S3 runs the UI, scanning, and BadUSB; the C6 handles deauthentication, beacon spam, BLE spam, and Zigbee sniffing, with the two processors communicating over a defined command/response protocol with auto-detection on boot. Currently functional: Wi-Fi scanner (2.4 and 5GHz, encryption type, RSSI), BLE scanner with OUI vendor lookup, passive 802.11 probe sniffer, BadUSB with Ducky script support, and the full co-processor attack suite. Hardware modules for sub-GHz (CC1101), NFC (PN532), and GPS are wired into the firmware with runtime hardware detection, so they activate on plug-in without reflashing. The RF Lab module combining sub-GHz and IR behind a rotary encoder and second display is the active build priority.

Tama Server Monitor Embedded

ESP32-S3 IoT monitor housed in a repurposed Tamagotchi shell, built to display real-time server telemetry on a 0.96" OLED. The server-side Python hub runs as a systemd service on an Orange Pi, polling server state and publishing pipe-delimited payloads over MQTT to a local Mosquitto broker. Pipe-delimited rather than JSON because parsing overhead on a constrained microcontroller is a real cost, not a theoretical one. The ESP32 subscribes to the broker and renders server name, status, player count, and TPS on the display. Deep sleep between updates cuts idle current from roughly 80mA to around 20 microamps, with a GPIO interrupt waking the device when a new message arrives. Getting meaningful information to fit and remain readable on a 0.96" OLED turned out to be a genuine UX constraint that shaped the layout decisions throughout. Full technical write-up published on Medium, covering the architecture, power design, and lessons from building within the constraints of a very small screen and a very small shell.
